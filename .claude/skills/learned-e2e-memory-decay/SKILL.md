## CRITICAL SAFETY RULE

**NEVER call `POST /reset` or `DELETE /forget` on port 8100.** Port 8100 is the PRODUCTION database with real user memories. Destructive operations will permanently delete data.

**For E2E testing:** Start a temporary server on a different port (e.g., 8199) with an in-memory or temp database:

```bash
.venv/bin/python -m memory_decay.server --port 8199 --db-path :memory: &
E2E_PORT=8199
# ... run all tests against $E2E_PORT ...
kill %1  # clean up temp server
```

Only read-only operations (health, stats, search) are safe against production.

## When to Use
- After deploying code changes to the memory-decay server
- After restarting the server process
- Before pushing to remote after server-side changes
- When user says "test it", "E2E test", "서버 테스트"

## Procedure

0. **Start temp server** — `.venv/bin/python -m memory_decay.server --port 8199 --db-path :memory: &`
1. **Health check** — `GET /health` returns `{"status": "ok"}`
2. **Reset** — `POST /reset` (safe: temp DB only)
3. **Store memories** — `POST /store` with Korean + English text, varying importance/mtype
4. **Store batch** — `POST /store-batch` with 3+ items
5. **Search (vector)** — `POST /search` with semantic query, verify results ranked by relevance
6. **Search (BM25)** — verify lexical overlap boosts ranking (if bm25_weight > 0 in params)
7. **Tick decay** — `POST /tick` with count=10, then search again — scores should decrease
8. **Consolidation** — search the same query twice, verify top result's retrieval_score increased (testing effect)
9. **Auto-tick** — `POST /auto-tick`, verify ticks_applied is returned
10. **Forget** — `DELETE /forget/{id}`, verify memory gone from search results
11. **Stats** — `GET /stats` verify num_memories matches expected count
12. **Kill temp server** — `kill %1` or by PID
13. **Verify production** — `GET http://127.0.0.1:8100/health` still ok
14. **Report** — print pass/fail summary table

## Known Failure Modes
- Server not running → health check fails, all tests skip
- Embedding model not loaded → first store/search slow (30s+), may timeout
- Port conflict → check if another process holds 8199 before starting temp server
- **Running destructive tests on port 8100** → PRODUCTION DATA LOSS

## Examples

**Pass criteria:**
- Health returns ok
- Stored memory appears in search top-3
- Scores decrease after tick
- Forgotten memory absent from results
- retrieval_score increases after repeated search (consolidation)
- Production server unaffected after test
